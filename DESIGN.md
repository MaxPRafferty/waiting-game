## Problem Statement

A web app with a single, impossibly long horizontal strip of checkboxes — potentially millions of them. Each user owns one checkbox, assigned at the position they join (back of the queue). They can scroll the strip to any position and see live state.

The mechanic:
- Users ahead of you in line check their own boxes, one by one from the front
- You can only check YOUR box when all boxes ahead of you are checked (or you're first)
- Leaving the tab removes you immediately; everyone behind you shifts forward
- Winning (checking your box when eligible) earns a global announcement + leaderboard entry
- New users pile in continuously behind you

Direct prior art: One Million Checkboxes (OMCB, June 2024) — shared state, anyone toggles anything. This is architecturally different: owned positions, sequential mechanics, liveness constraints. OMCB was a shared canvas; this is a distributed queue.

## What Makes This Cool

The number is the product. "#47,291 in line" is instantly shareable. Checkboxes are just the visual carrier.

The core mechanic creates compounding tension:
- You're simultaneously watching people ahead of you succeed and drop out
- You can see rage-quits happening in real time (ghost checkboxes fading)
- The longer you survive, the more you've invested — loss aversion kicks in hard
- Winning requires patience, not skill — which makes it democratic and infuriating in equal measure

The spectator angle (scrolling to arbitrary positions to watch other people wait) is an emergent social layer that costs nothing extra to support.

## Constraints

- TypeScript + Node.js throughout
- WebSockets for real-time state (no polling)
- Browser height limits (~17M px in Firefox) require custom scroll handling above ~500K checkboxes at 20px each
- Liveness detection via heartbeat — tab close = immediate removal
- No auth required; anonymous sessions via session token in localStorage

## Premises

1. When a user leaves (tab closed/navigate away), their slot is removed immediately and everyone behind shifts forward one position.
2. Users can scroll to any position in the queue and see live state there, even if that section is far from their own position.
3. The queue can theoretically grow to tens of millions of entries — no hard cap.
4. Position number (#N in line) is always relative to the front of the queue — dynamic, not fixed.

## Recommended Approach

**Approach B: Node.js + ws + Redis sorted set.**

The core design decision that makes this right:

**Separate queue order from displayed position.**

Each user gets an immutable **sequence number** (monotonically increasing integer) when they join. "Current position" is never stored — it's always computed as a live query: `count of sequence numbers < yours that are still connected`. This means:

- No fan-out problem: when user #N leaves, you don't broadcast "everyone shift by 1" to 50K connections
- Redis sorted set keyed by sequence number, with connection ID as value, handles this natively

**Range subscriptions for the viewport:**

Range subscriptions use **sequence number** as the key space, not dynamic position. When the client requests the viewport showing positions P through P+N (displayed to user), the server translates P→sequence number via a Redis `ZRANGE` rank query, then subscribes the client to sequence range [S, S+N].

The server maintains an **in-memory range→client map** (a sorted interval tree or simple bucketed map per Node.js process). On any queue mutation (join/leave/check), the server determines which sequence numbers are affected and pushes `range_update` events to relevant clients. This map lives in the Node.js process — if the server restarts, all range subscriptions are lost and clients must resubscribe (handled client-side on reconnect by re-sending `range_subscribe`).

**v1 scaling ceiling:** A single Node.js `ws` process handles ~10K–50K concurrent WebSocket connections on commodity hardware. This design is appropriate for v1. Horizontal scaling (multiple processes sharing a Redis pub/sub channel for cross-process range fan-out) is a v2 concern.

Viewport changes trigger `range_unsubscribe` for the old range + `range_subscribe` for the new range. Position display updates are **push-based only** — server sends `position_update` when the user's computed position changes due to upstream departures. No polling fallback.

**Range subscription fan-out (v1 implementation):** The server maintains an in-memory bucketed map: sequence space is divided into buckets of 1,000 slots each. Each bucket tracks which client connections are currently subscribed to that range. On any mutation affecting sequence N, the server looks up bucket `floor(N/1000)` and pushes updates to all subscribed clients in that bucket. This is O(subscribers_in_bucket) per mutation — adequate for v1 scale.

**Server restart consequence:** Because reconnects are explicitly rejected (no session token reuse), a server restart means all connected clients lose their place in the queue. Each client will receive a connection close event, reconnect, and be placed at the back. This is a known v1 limitation — deploy during low-traffic periods. Fly.io rolling deployments can mitigate but not eliminate this (the Node.js process restart is still a queue reset). Acknowledged and accepted for v1.

**Liveness via heartbeat:**

Client pings every 5 seconds. Server marks connection as stale after 12 seconds (two missed pings + buffer). Stale connection → immediately removed from Redis sorted set → server broadcasts `left` event to all clients currently subscribed to that sequence number's range.

**Ghost checkbox is purely cosmetic:** The slot is logically gone from the queue within 12-15 seconds (success criterion). The "ghost" visual (fading animation over 30s) is triggered by the `left` message on the client side — the server has already removed the slot. The 30-second fade is a frontend-only effect. Clients should not count ghost slots when computing positions.

**Virtual scroll:**

Since 10M checkboxes × 20px = 200M px (beyond browser limits), the scroll container uses a fixed height with a downscaled virtual scrollbar. Wheel events are intercepted to move a logical "offset" value, which drives which slice of the queue to fetch and render. At 20px per checkbox and a 1920px viewport width, ~96 checkboxes are visible — a DOM pool of 120 nodes (96 + buffer) is maintained and recycled as the offset changes. The scrollbar represents the full queue length via a scale factor: `scrollbar_pos / scrollbar_height = logical_offset / total_queue_length`.

## WebSocket Protocol

Minimum message schema (client ↔ server):

| Message | Direction | Payload |
|---------|-----------|---------|
| `join` | C→S | `{ session_token: string }` |
| `joined` | S→C | `{ sequence_number: number, position: number }` |
| `position_update` | S→C | `{ position: number }` (pushed on range change, not polled) |
| `viewport_subscribe` | C→S | `{ from_position: number, to_position: number }` — server translates to sequence range |
| `range_state` | S→C | `{ slots: Array<{ seq: number, position: number, state: 'checked' \| 'waiting' \| 'empty' }> }` |
| `range_update` | S→C | `{ seq: number, position: number, state: 'checked' \| 'waiting' \| 'empty' }` |
| `join_rejected` | S→C | `{ reason: 'rate_limited' }` |
| `ping` | C→S | `{}` (every 5s) |
| `pong` | S→C | `{}` |
| `check` | C→S | `{ session_token: string }` |
| `check_ok` | S→C | `{ position: number }` |
| `check_rejected` | S→C | `{ reason: 'not_eligible' \| 'already_checked' }` |
| `winner` | S→C (broadcast) | `{ username: string \| null, position: number, duration_ms: number }` |
| `left` | S→C (range broadcast) | `{ seq: number }` |

**Eligibility check is atomic:** server validates via Redis transaction (WATCH + MULTI/EXEC or Lua script) that `ZRANK(queue, session_token) === 0` (no entries with lower sequence number) before recording the check event. Race condition between two users claiming the same slot is impossible — only one can be at position 0.

## Open Questions

1. **Reconnect/rejoin:** Resolved — no. Session token is stored in localStorage for display purposes only. The server does not accept reconnects — a new tab means a new session token and a new queue position at the back. Deliberate cruelty is on-brand.
2. **Leaderboard permanence:** Resolved — yes. Winners are persisted to Redis with `SET winners:{seq} {json}` + `ZADD leaderboard {timestamp} {seq}`. The queue itself (active connections) is ephemeral.
3. **Ghost checkbox duration:** Resolved — 30 seconds, frontend-only. The slot is gone from the queue immediately; the ghost animation is cosmetic only.
4. **Maximum checkbox density:** Resolved as a prototype tunable. Start at 20px. If virtual scroll performance degrades, try 12px. Document the constant as `CHECKBOX_WIDTH_PX` in the frontend config.
5. **First user edge case:** Resolved — the first user to join (ZRANK === 0, no entries ahead) can check immediately. No holding state. The UI shows "You're first in line — check your box now!" instead of the standard countdown/waiting state. Rate-limit join events to 10/minute per IP to prevent griefing (user joins, checks, leaves, rejoins the front).
6. **Rate limiting:** IP-based join rate limit (10 joins/minute) implemented as a Redis counter with TTL. Soft block (connection rejected with `join_rejected: 'rate_limited'` message) rather than a hard ban.

## Success Criteria

- Two browser tabs: tab 1 checks, tab 2's checkbox activates in real time. Close tab 2, it disappears from tab 1's view. Full loop working.
- Scroll to an arbitrary position (e.g., #1,000 while you're at #50,000) and see live state there.
- Close a tab → slot removed within 15 seconds for all viewers.
- Position counter updates correctly as users ahead leave or check.
- Winner announcement appears globally across all open tabs within 1 second of a user checking.
- Screenshot of "#47,291 in line" is genuinely shareable without context.

## Distribution Plan

- **Hosting:** Fly.io or Railway (Node.js + Redis in one service). Not Vercel — this is stateful, long-lived WebSocket connections don't fit serverless well.
- **Redis:** Fly.io Redis addon (colocated, TCP, zero extra round-trip latency). Not Upstash HTTP mode — 2K Redis ops/second from heartbeats alone makes per-request HTTP latency unacceptable. Upstash TCP mode is acceptable if Fly.io Redis is unavailable.
- **Domain:** Something memorable. `waitinggame.lol` or similar.
- **No CI/CD for now:** Manual `fly deploy` until the queue is stable. Add GitHub Actions after v1.
- **OG image generation:** Post-v1 nice-to-have. Server-side canvas or `@vercel/og` to generate a per-user shareable image showing their position number. Not required for v1 — users can screenshot.
- **Rage-quit counter:** Post-v1. Requires a Redis counter (`INCR ragequits:today` with daily TTL). Not a correctness feature; add after core mechanic is live.
